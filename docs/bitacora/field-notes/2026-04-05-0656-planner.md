## Contexto
Se evaluó si la spec `docs/specs/catalogo-turnos-crear-turno.md` debía desglosarse o implementarse como un solo issue. Se decidió desglosar en 3 issues secuenciales.

## Descubrimientos
- La complejidad real está en los value objects de franja (semántica temporal con offsets) y en el factory del evento TurnoCreado (árbol de contención + acumulación de errores)
- El aggregate y handler son wiring mecánico una vez que las piezas anteriores existen

## Decisiones
- **Desglose en 3 issues secuenciales**: VOs de franja (#2) → evento TurnoCreado con factory (#3) → aggregate + handler + endpoint (#4). Epic padre: #5
- **Acumulación de errores**: `AggregateException` con `ArgumentException` por error individual — no Result pattern, no FluentValidation
- **Value objects en Contracts**: FranjaOrdinaria, FranjaDescanso, FranjaExtra van en Contracts porque otros dominios los necesitarán
- **Evento TurnoCreado privado**: vive en el dominio Programación por ahora. DTO compartido en Contracts vendrá cuando haya consumidores externos

## Descartado
- Issue único para toda la feature — demasiado grande para un ciclo TDD
- Result pattern para errores del factory — AggregateException es suficiente dado que el handler de Wolverine ya maneja excepciones

## Preguntas abiertas
- ¿Límite máximo de franjas por turno?
- ¿Validar duración máxima legal del turno o eso es de CálculoHoras?
- ¿ToString() en el aggregate o en un value object Turno?
