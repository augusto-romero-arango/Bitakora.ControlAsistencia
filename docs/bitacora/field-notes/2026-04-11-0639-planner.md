## Contexto
El usuario quiere usar la cobertura como herramienta de deteccion de brechas en los agentes de codificacion, no como metrica de vanidad. Prefiere branch coverage sobre line coverage.

## Descubrimientos
- El profiler dinamico de `dotnet-coverage` no funciona con .NET 10 en macOS ARM64. Solo la instrumentacion estatica (`dotnet-coverage instrument` antes de `collect`) produce datos reales.
- Las clases auto-generadas por el SDK de Azure Functions (DirectFunctionExecutor, WorkerExtensionStartupCodeExecutor, etc.) inflan el denominador: bajan ControlHoras de ~85% real a 54.5% reportado.
- La logica de dominio (aggregates, handlers, VOs, validators) esta consistentemente al 100%. Los agentes hacen bien su trabajo en el core.
- La unica brecha real de dominio es el FunctionEndpoint del ServiceBus trigger en ControlHoras (0% cobertura), que tiene logica de orquestacion de errores no testeada.
- Branch coverage no se puede medir actualmente — la instrumentacion estatica no lo reporta.

## Decisiones
- Crear issue de tooling (#56) para mejorar la medicion en CI: instrumentacion estatica, exclusion de codigo generado, evaluacion de branch coverage.
- Crear issue de feature (#57) para testear el FunctionEndpoint del ServiceBus trigger en ControlHoras — la unica brecha genuina.
- No perseguir cobertura de RequestValidator, HealthCheck ni Program.cs — es boilerplate sin valor de testing unitario.
- Crear issue de tooling (#58) para agregar un coverage gate al pipeline TDD como Stage 4, con loop acotado de remediacion (1 iteracion max).

### Diseno del coverage gate (decision clave)
Se evaluaron 3 opciones:
- **A: Reviewer absorbe la medicion** — descartada porque mueve la responsabilidad al mismo nivel del gap, sin ganar visibilidad.
- **B: Stage separado con loop acotado** — elegida. Gate mecanico que mide, clasifica, remedia (1 vez) y reporta.
- **C: Cobertura como contexto informativo** — descartada porque depende del juicio inconsistente del LLM reviewer.

Clasificacion de archivos por umbral:
- Logica (95%+): CommandHandler, AggregateRoot, Validator, Eventos con factory, VOs con invariantes, FunctionEndpoint
- Excluidos: HealthCheck, Program, Mensajes, AssemblyMarker, ConfiguracionSerializacion, records DTO puros, Infraestructura de wiring

## Descartado
- No se crearon issues para los 2-3% faltantes en FranjaOrdinaria y SubFranja — son paths de serializacion/igualdad que no representan riesgo de negocio.
- No se considero coverlet como alternativa — tiene problemas similares con .NET 10 y xUnit v3 MTP.
- Opcion A (reviewer mide cobertura) descartada por falta de separacion de responsabilidades.
- Opcion C (cobertura como info pasiva) descartada por inconsistencia del juicio LLM.

## Preguntas abiertas
- Cuando dotnet-coverage soporte branch coverage con instrumentacion estatica en .NET 10, ajustar umbrales del coverage gate (#58).
- El CI probablemente esta produciendo reportes vacios por el mismo problema del profiler. Verificar en el proximo PR.
- El test-writer en modo patch escribe tests que probablemente pasen de una (la implementacion ya existe). Evaluar si el gate post-patch debe ser "tests pasan" en vez de "tests fallan".
