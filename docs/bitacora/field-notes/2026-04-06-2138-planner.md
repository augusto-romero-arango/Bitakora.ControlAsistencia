## Contexto
El usuario queria crear un endpoint para registrar la programacion de un turno a un empleado. La conversacion evoluciono hacia descubrimientos profundos sobre la arquitectura de dominios.

## Descubrimientos

### Programacion como traductor de intenciones
- Programacion no es solo un catalogo de turnos ni un relay. Su rol es traducir intenciones humanas (rangos de fechas, ciclos, patrones) en eventos atomicos diarios que ControlHoras consume.
- El bloque atomico es: un empleado, N dias, un turno. Construcciones complejas (semanarios, ciclos) se reducen a este mismo comando.
- Programacion resuelve el turno del catalogo internamente. El comando recibe TurnoId, el evento sale con el turno completo.

### Nacimiento de ControlHoras
- Depuracion y CalculoHoras no son dominios distintos — son un solo dominio: **ControlHoras**.
- ControlHoras recibe programaciones y marcaciones, depura, discrimina horas, y controla transmision a nomina.
- El aggregate **ControlDiario** representa el dia de trabajo de un empleado (identidad: EmpleadoId + Fecha).

### Solicitud vs programacion directa
- El comando se llama SolicitarProgramacionTurno (no ProgramarTurno) porque es una solicitud — ControlHoras es quien la materializa.
- El aggregate SolicitudProgramacionAggregateRoot persiste la intencion original con trazabilidad (correlationId).
- No hay aggregate intermedio de programacion diaria en el dominio Programacion.

### Representaciones para eventos
- DetalleTurno, DetalleFranjaOrdinaria, DetalleSubFranja: representaciones planas del turno para eventos. No son DTOs — son el lenguaje natural del dominio para la comunicacion entre bounded contexts.
- InformacionEmpleado: datos del empleado que viajan en eventos. No se llama DetalleEmpleado porque "la informacion del empleado" es mas natural.
- No hay convencion mecanica de prefijo (Detalle* vs Informacion*). Cada nombre se elige por naturalidad.

### EmpleadoId
- Es un string opaco decidido por el cliente (puede ser cedula, guid, numero de contrato).
- Vive como propiedad de InformacionEmpleado, no como campo separado.
- Permite que un mismo empleado recontratado tenga identidades distintas.

### Dos endpoints, no uno
- ProgramarTurnoDelCatalogo: recibe TurnoId, Programacion resuelve el turno.
- ProgramarTurnoAlVuelo: recibe estructura completa, no se guarda en catalogo.
- Son dos comandos distintos, dos issues distintos. Mismo evento de salida.

## Decisiones
- Dominio nuevo: ControlHoras (reemplaza Depuracion + CalculoHoras)
- Aggregate: ControlDiario (EmpleadoId + Fecha)
- Label creado: dom:control-horas
- No se crean epicas — el usuario no ve valor en ese tipo de issues
- El issue de scaffold (#11) va separado del feature (#12)

## Descartado
- Aggregate intermedio de ProgramacionDiaria — no aporta valor, duplica historia
- Consistencia mecanica de prefijos para representaciones de eventos (Detalle* vs Informacion*)
- EmpleadoId como campo separado fuera de InformacionEmpleado

## Preguntas abiertas
- Features futuros de ControlHoras: marcaciones, depuracion, discriminacion
- SolicitarProgramacionTurnoAlVuelo (segundo endpoint)
- Renombrar dominio Programacion (se queda o se ajusta?)
- Fases del ControlDiario: Programado, Marcado, Depurado, Discriminado, Transmitido — como se modelan?
