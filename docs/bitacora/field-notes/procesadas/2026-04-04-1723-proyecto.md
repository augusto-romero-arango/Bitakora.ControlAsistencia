## Contexto

Sesion de diseno del flujo de implementacion. Se partio de una necesidad concreta: el skill `/implement` no detectaba si un issue requeria un dominio nuevo, y no habia garantia de que los issues llegaran completos al pipeline TDD.

## Descubrimientos

- El test-writer depende criticamente de dos secciones del issue: **Modelo de eventos** (para nombrar comandos, eventos, aggregates) y **Criterios de aceptacion** (para generar tests). Sin ellas trabaja a ciegas.
- El implementer es el menos dependiente del issue — se guia por los tests. El issue es contexto secundario.
- El reviewer necesita los CAs para validar cobertura con su tabla `Criterio | Estado | Test(s)`.
- El modo `desglosar` del planner creaba issues directamente como `estado:listo`, sin garantia de completitud individual.

## Decisiones

- **Definition of Ready (ADR-0014)**: tabla de criterios por tipo de issue (`feature`, `refactor`, `bug`, `infra`, `tooling`). Es el contrato entre planeacion y codificacion.
- **Scaffold en worktree**: si un dominio no existe, el scaffold se hace en el mismo worktree del issue (Stage 0 del pipeline). El PR resultante es atomico: infraestructura + primera feature.
- **Scaffold requiere confirmacion**: nunca autonomo — implica Terraform + GitHub Actions.
- **Desglosar crea borradores**: los sub-issues nacen como `estado:borrador` con cuerpo enriquecido. Se refinan individualmente antes de ir a desarrollo.
- **Explorar respeta el DoR**: solo crea `estado:listo` si el knowledge crunching produjo toda la info. Si no, crea borrador.
- **`/implement` valida DoR**: defensa en profundidad — verifica labels y secciones del body antes de lanzar el pipeline.
- **domain-scaffolder se mantiene como agente** (no skill): los skills no pueden invocar otros skills, pero si pueden invocar agentes.

## Artefactos producidos

- `docs/adr/0014-definition-of-ready.md` — ADR con tabla DoR y reglas de validacion
- `.claude/agents/planner.md` — modos explorar, desglosar y refinar ajustados
- `.claude/commands/implement.md` — validacion DoR (paso 1.5) + deteccion de dominio (pasos 2-3)
- `scripts/tdd-pipeline.sh` — flag `--scaffold-domain` y Stage 0
- `scripts/tmux-pipeline.sh` — passthrough del flag

- Los issues de tipo `tooling` no requieren `dom:X`. Si un tooling afecta un dominio especifico, deberia llevar el label?
