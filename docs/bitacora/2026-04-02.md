**Control de costos de Application Insights en 4 capas (19:47)**
Motivado por un incidente real de $350 en 3 dias (ver seccion Problemas), se implemento:
1. **Log levels en host.json**: default Warning, solo codigo de negocio en Information, Marten y Wolverine explicitamente en Warning
2. **Sampling**: 5 items/seg (vs 20 por defecto). Excepciones nunca se muestrean
3. **Daily Cap en Terraform**: 0.5 GB/dia (~$35/mes maximo)
4. **Alertas**: ingestion al 80% del cap (cada hora) + pico de excepciones >50 en 5 min

## Problemas encontrados

**`terraform.tfvars` con `subscription_id` en el repositorio (12:17)**
Al crear el modulo de PostgreSQL, el archivo `.tfvars` quedo trackeado en git con el subscription_id. Se detecto antes de hacer push. Se removio del tracking, se paso a GitHub Secrets (`TF_VAR_subscription_id`, `TF_VAR_postgresql_admin_password`) y se agrego `*.tfvars` a `.gitignore`.

**Incidente de $350 en Application Insights (descubierto en algun momento del dia)**
Una funcion Azure con errores habia generado millones de registros en 3 dias en el proyecto de referencia, resultando en un costo completamente inesperado. El sistema no tenia daily cap, ni alertas de costos, ni configuracion de log levels. Marten y Wolverine generan logs `Information` extremadamente verbosos durante sus mecanismos de retry internos. Esto motivo la estrategia de 4 capas documentada en ADR-0009.

**PostgreSQL bloqueado en eastus2 por `LocationIsOfferRestricted` (20:25)**
Al hacer `terraform apply`, Azure rechazo la creacion del PostgreSQL Flexible Server en `eastus2` con error de restriccion regional. No habia ningun warning previo en la documentacion de Terraform ni en el portal. Se migro a `centralus` como alternativa.

**Nombre del PostgreSQL reservado tras el intento fallido (commit del 3 de abril, 08:31)**
Despues de migrar a `centralus`, Azure rechazo el nombre `psql-controlasistencias-dev` porque habia quedado reservado en `eastus2` tras el intento fallido anterior. Azure no permite reusar nombres de PostgreSQL Flexible Server que estuvieron en proceso de creacion, incluso si la creacion fallo. Se introdujo una variable `project_short` y un local `prefix_short` para centralizar un nombre corto; el servidor paso a llamarse `psql-asist-dev`.

**State locks huerfanos en CI/CD de Terraform (19:37)**
Las ejecuciones paralelas de los workflows de Infra CI y Infra CD operaban sobre el mismo state, generando locks huerfanos que bloqueaban ejecuciones posteriores. Se agrego un `concurrency group` compartido entre ambos workflows y `-lock-timeout=5m` como proteccion adicional.

**Conflicto de merge en domain-scaffolder.md (19:08)**
Al hacer merge del tracking remoto despues de trabajar en la rama del PR #1, hubo un conflicto en `.claude/agents/domain-scaffolder.md` que requirio resolucion manual.

**Secciones duplicadas en domain-scaffolder (19:26)**
El scaffolder tenia bloques repetidos de configuracion Marten/Wolverine/OTel, paquetes ES duplicados con versiones inconsistentes (0.1.*), y una mencion huerfana a AssemblyMarker. Producto de ediciones incrementales sin refactor. Se consolido y renumero todo.

## Lo que descartamos

**ADR-0004 original: topic por dominio con filtros SQL**
El modelo del dia anterior (topic por dominio productor, subscriptions con SQL filter) se marco como obsoleto. La razon: para saber quien consume que evento habia que leer las reglas de filtro SQL de cada subscription, complejidad operativa completamente oculta. Ademas, agregar un nuevo tipo de evento al dominio no requeria crear infraestructura, lo que podia llevar a eventos fantasma sin consumidores sabiendolo. MassTransit usa topic-per-event como su topologia por defecto, validando la nueva decision.

**CRUD tradicional para persistencia de dominio**
Descartado a favor de Event Sourcing. Razon concreta: el sistema calcula horas y liquidacion de nomina, lo que requiere un log de auditoria completo e inmutable. Con CRUD no se puede responder "que paso exactamente el martes a las 6am" sin logs adicionales que son ciudadanos de segunda clase.

**Implementar manualmente el flujo de comandos/eventos**
Habia la opcion de implementar el patron CommandHandler -> AggregateRoot -> eventos sin librerias especializadas. Se descarto porque los paquetes `Cosmos.EventSourcing.*` ya estan validados en produccion en ControlPlane con el mismo stack.

**Result Pattern entre CommandHandler y Endpoint HTTP**
Se considero usar un Result<T> entre el handler y el endpoint. Se descarto porque es innecesario: el endpoint HTTP siempre responde 202 si la validacion inicial paso; el procesamiento del dominio es asincrono. El aggregate emite eventos de fallo (no excepciones) para errores de negocio.

**NSubstitute para mocking en tests ES**
Se evaluo NSubstitute para mockear dependencias en los tests de command handlers. Se descarto a favor de fakes manuales con comportamiento predecible. El harness `Cosmos.EventSourcing.Testing.Utilities` ya provee el DSL necesario; agregar un framework de mocking encima creia capas de indireccion innecesarias.

**Un servidor PostgreSQL por dominio**
Descartado por costo y complejidad operacional excesivos para la fase de desarrollo. La estrategia es un servidor compartido con schemas separados por dominio, aceptando como deuda tecnica que la contrasena viaja como variable de Terraform sensible (no via Key Vault references).

**GitHub Projects para gestion de tareas**
Descartado por overhead excesivo para un solo developer en fase de exploracion. Las task lists nativas de GitHub en issues epic cubren la necesidad de task graph sin crear una herramienta adicional que mantener.

**Milestones de GitHub**
Descartados porque implican planificacion temporal que no aplica en esta fase del proyecto. El numero del issue es suficiente como identificador.

**Prefijos en titulos de issues (EMP001, feat:, HU-)**
Descartados porque los labels dimensionales cumplen la funcion de categorizar. Los prefijos en titulos son redundantes y dificultan leer el backlog.

**Modelo de eventos en archivo YAML separado (eda-modeler)**
El skill `eda-modeler` existente crea archivos YAML de flujos de eventos. Se evaluo usarlo como fuente de verdad para el modelo de dominio en los issues. Se decidio que es complementario, no alternativo: el modelo de eventos dentro del issue es mas compacto y especifico a una tarea.

## Aprendizajes

1. **Azure retiene nombres de recursos tras intentos fallidos.** Un `terraform apply` que falla a mitad de creacion de un PostgreSQL Flexible Server deja el nombre reservado en la region original. La solucion no es solo cambiar de region sino tambien cambiar el nombre del recurso.

2. **Las restricciones regionales de Azure no estan siempre documentadas de antemano.** `LocationIsOfferRestricted` en eastus2 para PostgreSQL Flexible Server se descubrio durante el apply, no antes. Vale la pena verificar disponibilidad de recursos criticos en la region objetivo antes de escribir el modulo.

3. **Terraform en CI/CD requiere concurrency groups explicitos.** Sin ellos, ejecuciones paralelas de CI y CD sobre el mismo state generan locks huerfanos. El pattern es: un grupo compartido entre todos los workflows que tocan el mismo state, con CI cancelando runs anteriores y CD nunca cancelando.

4. **Application Insights sin limites es una bomba de costos con mecha corta.** Los defaults (sin daily cap, sampling de 20 items/seg, todo en nivel Information) pueden generar cientos de dolares en dias si una funcion entra en loop de errores. La estrategia de 4 capas es defensiva y debe aplicarse desde el dia cero.

5. **Los paquetes de telemetria sin uso activo crean riesgo latente.** `Azure.Monitor.OpenTelemetry.AspNetCore` estaba en el csproj sin que `UseAzureMonitor()` se invocara en ninguna parte. Si alguien lo activaba sin configurar sampling, generaria doble ingestion. Se removio.

6. **El domain-scaffolder acumula deuda tecnica rapido con ediciones incrementales.** Despues de 4-5 iteraciones ya tenia secciones duplicadas y numeracion rota. Un refactor de consolidacion periodico es necesario para que siga siendo usable.

7. **Nombrar comandos, eventos y aggregates antes de codificar estabiliza el vocabulario.** La sesion de Knowledge Crunching con el planner produce un modelo de eventos que es input directo para `es-test-writer`. Cuando el vocabulario se deja implicito, los agentes de codificacion lo inventan y despues hay que renombrarlo.

8. **`es-reviewer` usa opus, no sonnet.** La decision fue explicita: el reviewer es el guardian final de calidad del aggregate y el command handler. Merece el modelo mas capaz disponible.
