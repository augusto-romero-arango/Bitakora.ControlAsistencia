## Problemas encontrados

**El timeout del implementer y los 95 minutos perdidos (issue #10 — 10:55)**
El pipeline arranco el implementer para el issue #10 a las 10:55. A las 12:48 el watchdog disparo `kill -9 $PID` — pero mato el subshell, no el proceso `claude` hijo. El proceso siguio corriendo como huerfano. A los 95 minutos el sistema lo mato por signal 137 (SIGKILL del OS).

Lo ironico: el implementer *si habia terminado su trabajo*. El commit existia en el worktree con 109/109 tests pasando. El pipeline aborto sin saberlo. Recovery manual: rebase sobre main, revision del reviewer, creacion del PR #14 a mano. Costo estimado: ~2 horas de tiempo de computo mas 1 hora de trabajo manual.

**Bug del kill — process group vs. subshell (commit 6469f05)**
La causa del timeout descontrolado: `kill -9 $PID` sobre un subshell `(cd && claude ...)` no propaga la señal al proceso hijo. Fix: `kill -9 -$PID` mata el process group completo. Un caracter de diferencia, dos horas de diferencia.

**`[JsonConstructor]` no funciona con Marten (PRs #15 y #16)**
`TurnoCreado` tenia un constructor privado parametrizado (para el factory) y un constructor vacio privado (para STJ). El plan documentado en el ADR-0015 y en los agentes era que STJ usara el constructor vacio via reflection.

Lo que no sabiamos: Marten no usa STJ directamente para deserializar eventos. Tiene su propia capa de serializacion. `[JsonConstructor]` es un atributo de STJ — Marten lo ignora. El PR #15 agrego el atributo y los tests pasaron en CI... pero la deserializacion en Marten seguia rota.

El PR #16 aplico el patron correcto: `ConfigurarSerializacion` registrado en `Program.cs` para que Marten sepa como construir el objeto. El mismo patron que ya usabamos para los VOs con invariantes, pero que no habiamos aplicado consistentemente a los eventos.

**Pipeline fallo nuevamente en stage 1 del issue #12 (22:43)**
El test-writer del issue #12 no produjo tests fallando — exit code 1, misma causa que episodios anteriores: el agente escribio implementacion real en lugar de stubs. El pipeline aborto. Se corrigio manualmente.

## Lo que descartamos

**Pausa/resume mid-pipeline para intervencion humana**
Se considero que el implementer bloqueado pidiera intervencion del humano y luego retomara. Descartado por complejidad: requeria un mecanismo de pausa/resume que no existe, y el reviewer como segundo intento ofrece el mismo beneficio con menor friccion. Documentado en la field note del incidente.

**`[JsonConstructor]` como solucion para Marten**
Tecnicamente correcto para STJ puro. Insuficiente para Marten. Se descarto despues del PR #15 — el atributo no es la interfaz que Marten usa para la construccion de objetos.

## Aprendizajes

1. **`kill -9 $PID` no mata procesos hijos en subshells.** El watchdog debe usar `kill -9 -$PID` (signo negativo = process group). Con `$PID` positivo, el proceso hijo queda huerfano y el timeout de 30 min se convierte en 95 min de OS timeout.

2. **Marten no delega en STJ para deserializar eventos.** `[JsonConstructor]` es un contrato con el serializador de System.Text.Json, no con Marten. Para eventos con constructores privados, el registro `ConfigurarSerializacion` en `Program.cs` es obligatorio — exactamente igual que para los VOs con invariantes. La regla del ADR-0015 aplica a todos los objetos que Marten necesita reconstruir, no solo a los VOs.

3. **Un pipeline que aborta sin PR no es silencioso — es opaco.** Cuando el implementer termina su trabajo pero el stage de verificacion falla, se pierde visibilidad de que el trabajo *si ocurrio*. La deteccion de bloqueo y la creacion del PR con estado parcial resuelve esto: el humano puede ver exactamente hasta donde llego el agente.

4. **La recovery manual de un worktree es viable y documentable.** Cuando el pipeline aborta pero el worktree tiene commits utiles, el flujo manual (rebase, reviewer, PR) tarda ~1 hora pero produce el mismo resultado. Vale la pena documentarlo en la field note para no improvisar cada vez.

5. **El mismo bug puede requerir dos PRs cuando la primera solucion parece correcta pero falla en un nivel de abstraccion diferente.** `[JsonConstructor]` pasa los tests unitarios (que usan STJ directamente) pero falla en integracion con Marten (que tiene su propia capa). Los tests unitarios de serializacion no eran suficientemente integradores.
