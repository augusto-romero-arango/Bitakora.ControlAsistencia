# 2026-04-04 - El dia que el proyecto aprendio a pensar sobre si mismo

> Un dia sin una linea de codigo de negocio: todo el trabajo fue infraestructura de conocimiento — los agentes que facilitaran las sesiones de dominio, los artefactos que consumiran, y las reglas que garantizaran que el trabajo llegue completo al pipeline TDD.

## Lo que se logro

**Sistema de field notes y agentes de memoria (06:53 — commits 63ca6be, 8915793)**
El proyecto tenia una bitacora (`docs/bitacora/`) pero sin mecanismo de alimentacion sistematico. Hoy se construyo el ciclo completo:
- Agente `event-stormer` (opus): facilitador de sesiones de descubrimiento con field notes obligatorias al cerrar
- Agente `historiador` (sonnet): lee field notes + git + issues y produce la entrada diaria de bitacora, con dialogo previo al usuario antes de escribir
- Field notes en `docs/bitacora/field-notes/` como artefacto de sesion; las procesadas se mueven a `field-notes/procesadas/`
- La primera field note del proyecto (8915793) documenta la propia sesion que diseno el sistema — meta-recursividad intencional

**Consolidacion del pipeline TDD (commits 9a269ca, ad0ec51, 0a16cdf)**
Los agentes `es-test-writer`, `es-implementer` y `es-reviewer` se consolidaron en los agentes estandar `test-writer`, `implementer` y `reviewer`. El razonamiento: mantener dos familias paralelas creaba deuda de sincronizacion sin ganancia real — todos los dominios del proyecto son Event Sourcing por arquitectura. Ademas:
- Soporte tmux para monitoreo visual de pipelines paralelos (9a269ca)
- Los agentes workflow (`dev-workflow`, `batch-workflow`, `parallel-workflow`) se reemplazaron por los skills `/implement`, `/batch` y `/parallel` — mas descubribles y sin necesidad de invocar claude con `--agent`

**ADR-0012: Patron de mensajes con .resx per-aggregate (commit 7acfe95)**
Se formalizaron las convenciones para que los mensajes de validacion y error vivan en archivos `.resx` por aggregate con una clase `Mensajes` anidada. Sin esto, los mensajes quedaban embebidos en los command handlers como strings literales dispersos.

**Knowledge hub `docs/eda/` y pipeline de conocimiento del dominio (commit 6a7cb67)**
Motivado por la frustracion concreta con la primera sesion del `event-stormer` (ver Problemas), se creo el hub de conocimiento del dominio:
- `ubiquitous-language.yaml`: terminos, actores con responsabilidades y frustraciones, sistemas externos, preguntas abiertas
- `context-map.yaml`: bounded contexts con tipo de relacion DDD y semantica de cada conexion
- `aggregates/`: un YAML por aggregate con estado, invariantes, comandos y downstream policies
- `catalog.yaml`: catalogo de eventos y comandos del sistema
- ADR-0013 documenta las decisiones de arquitectura de este pipeline

## Problemas encontrados

**El agente `event-stormer` era sordo a fuentes externas (field note 14:50)**
La primera sesion de lenguaje ubicuo evidencio un problema de diseno: el agente tenia una allowlist de tools que excluia WebSearch y WebFetch. Respondia desde el entrenamiento del modelo, no desde fuentes verificadas. Para un agente que debe investigar terminologia DDD, tecnicas de Event Storming y contexto legal colombiano, esto era inaceptable.

El segundo problema: el unico artefacto del agente eran las field notes (narrativa libre). El planner no tenia forma eficiente de consumir ese conocimiento — tenia que leer prosa para extraer datos. La creacion del knowledge hub `docs/eda/` con YAMLs estructurados resolvio esto, pero costo media sesion de trabajo diagnosticar el problema y disenar la solucion.

**El modo `desglosar` del planner creaba issues directamente como `estado:listo`**
Sin garantia de completitud. El test-writer y el reviewer dependen criticamente de secciones especificas del issue (Modelo de eventos, Criterios de aceptacion). Un issue que llegaba sin esas secciones hacia que el test-writer trabajara a ciegas. La solucion fue que `desglosar` cree borradores, no issues listos — el refinamiento es un paso separado.

## Lo que descartamos

**Reemplazar plan mode completamente con el agente proyecto**
El usuario usaba plan mode nativo para cosas rapidas y queria mantenerlo. La separacion "plan mode para lo tactico / agente evento-stormer para sesiones de descubrimiento significativas" resolvio el problema sin over-engineering. Clave: al aclarar que plan mode es "instrucciones bien escritas + restriccion de sistema" (no magia del modelo), el miedo a que un agente custom fuera inferior se disipo.

**Hook que escribe field notes automaticamente al salir de plan mode**
Imposible tecnico: los hooks `PostToolUse` reciben metadata de tool calls, no el contenido de la conversacion. Un hook que dispara al final de la sesion no tiene acceso a lo que se discutio. El mecanismo de recordatorio pasivo (un `echo` en el hook) fue el maximo alcanzable.

**Skill `/bitacora` para generacion rapida de bitacora**
Descartado en favor del agente `historiador` conversacional. La bitacora no es un changelog — requiere dialogo con el usuario para capturar contexto verbal que no esta en ningun archivo ("hoy fue frustrante porque...", "lo mas importante fue cuando..."). Un skill que ejecuta sin dialogo no puede capturar eso.

**Artefactos de dominio en Markdown estructurado**
Para el knowledge hub, se evaluo Markdown con frontmatter. Se eligio YAML porque es mas parseable para los agentes — pueden leer solo la seccion `invariants` de un aggregate sin procesar prosa. Los agentes consumen datos, no narrativa.

**Un artefacto monolitico para todo el dominio**
Un solo archivo YAML con todo el dominio. Se descarto porque cada agente tiene responsabilidades distintas: el `event-stormer` actualiza el glosario, el `eda-modeler` actualiza los flujos, el `planner` lee los aggregates. Separar por tipo permite que cada agente actualice solo su parte sin riesgo de colision.

**Dar WebSearch al `eda-modeler`**
El usuario lo rechazo explicitamente: investigar es rol del `event-stormer`, el `eda-modeler` solo modela lo ya descubierto. La separacion de roles es intencional.

**Herramientas externas (Notion, Miro) para el knowledge hub**
Rompen el flujo de terminal y no son versionables con git. El hub en `docs/eda/` vive donde vive el codigo.

**Los sub-issues del modo `desglosar` como `estado:listo` directos**
Descartado en favor de borradores. Ver Problemas para el detalle.

**`domain-scaffolder` como skill**
Los skills no pueden invocar otros skills, pero si pueden invocar agentes. El scaffolder necesita coordinacion compleja (Terraform + GitHub Actions + confirmation). Se mantiene como agente.

## Aprendizajes

1. **Las allowlists de tools en agentes tienen consecuencias no obvias.** Cuando un agente define `tools:` en su frontmatter, las herramientas MCP no se heredan automaticamente. Un agente investigador sin WebSearch/WebFetch es un agente que solo puede usar su entrenamiento — un problema grave para dominios donde el contexto local importa (legislacion colombiana, convenciones del equipo).

2. **El razonamiento del agente vale mas que su output.** La sesion de lenguaje ubicuo de las 14:02 fue valida no por las definiciones que produjo sino por las alternativas que descartos y el razonamiento detras. Capturar ese razonamiento en field notes estructuradas es mas valioso que capturar solo los terminos finales.

3. **Los artefactos de conocimiento deben diseñarse para su consumidor.** Las field notes (narrativa) son para el historiador y el usuario. Los YAMLs de `docs/eda/` son para los agentes del pipeline. Cuando el artefacto no esta disenado para quien lo consume, el consumidor trabaja mas de lo necesario.

4. **El Definition of Ready es un contrato, no una burocracia.** La tabla de criterios DoR (ADR-0014) define que necesita cada tipo de issue para que el pipeline TDD funcione bien. Sin ese contrato explicito, el pipeline falla en silencio — el test-writer genera tests de baja calidad, el reviewer no puede validar cobertura real.

5. **Los tokens tienen costo en cada inicio de sesion.** El cuadro de bienvenida en CLAUDE.md era visualmente bonito pero se leyendo en cada sesion. La heuristica: si la informacion ya esta en CLAUDE.md completo, duplicarla en un cuadro solo suma tokens sin sumar contexto.

6. **Jornada vs Turno es una distincion que importa en codigo.** En Colombia, "jornada" es el concepto juridico (duracion maxima, limites legales). "Turno" es el operativo (de 7AM a 5PM el lunes). Nombrarlos igual en el modelo de dominio seria un bug semantico latente que los tests no atraparian.
