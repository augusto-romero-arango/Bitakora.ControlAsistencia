# 2026-04-05 - El pipeline aprobó. El diseño, no.

> Un dia de arco completo: diseño de dominio, pipeline TDD, PR abandonado deliberadamente y reescritura desde cero con nuevos conceptos — y el aprendizaje de que en vibe coding hay un umbral donde corregir cuesta mas que tirar.

## Lo que se logro

**Diseño del evento TurnoCreado y el modelo de franjas (06:33 — commit a6a001c)**
Sesion de event-stormer que produjo el modelo central del feature "Crear Turno". Los descubrimientos clave:
- El modelo de *contencion*: una FranjaOrdinaria contiene FranjasDescanso y FranjasExtra. Descansos y extras son hojas — no pueden anidarse mas. Las franjas de un mismo padre no se solapan.
- *Semantica inicio-inclusivo, fin-exclusivo*: una franja de 8:00 a 11:00 tiene exactamente 180 minutos. Las 11:00 no se cuentan. Permite franjas contiguas sin ambiguedad.
- *Offset +1 en lugar de hora de corte*: para turnos que cruzan medianoche, se usa un offset de dia relativo (+0 implicito, +1 explicito). Se descarto la idea de "hora de corte" porque la particion legal por medianoche es responsabilidad de CalculoHoras.
- *Comando plano, factory rico*: el comando es tres arreglos de tuplas (ordinarias, descansos, extras) sin campo "tipo". El factory del evento construye el arbol de contencion, infiere los offsets cuando fin < inicio, y acumula todos los errores de validacion antes de lanzar una sola excepcion.

**Desglose y refinamiento del feature Crear Turno (07:13 — commit 614880f)**
El planner desgloso la spec en tres issues secuenciales refinados a `estado:listo`:
- `#2` Value objects de franja temporal (21 criterios de aceptacion)
- `#3` Evento TurnoCreado con factory (17 CAs) — bloqueado por #2
- `#4` Aggregate + handler + endpoint HTTP (8 CAs) — bloqueado por #3
- `#5` Epic padre "Crear turno en el catalogo de turnos"

Decisiones de la sesion del planner: `AggregateException` para acumulacion de errores (no Result pattern ni FluentValidation), VOs en `Contracts` porque otros dominios los necesitaran, `TurnoCreado` privado del aggregate por ahora (sin consumidores externos).

**ADR-0015: Heuristicas de modelado de objetos de dominio (commit 08db91c)**
Resultado de las sesiones de diseno de la tarde. La tabla central:

| Tipo de objeto | Forma | Constructor | Validacion |
|---|---|---|---|
| Comando / Evento sin invariantes | `record` | Publico | Externa |
| VO con invariantes / Evento con precondiciones | `sealed class` + factory static | Privado x2 | Factory → throw |
| AggregateRoot | `partial class` | Factory `Crear` | Evento de fallo (ADR-0007) |

La distincion critica documentada: el criterio no es "tiene reglas" sino **mutabilidad**. Un VO puede tener validaciones y seguir siendo un record inmutable; si nunca muta despues de crearse, `record` es correcto. Si encapsula estado privado con invariantes, `sealed class`.

**Patron de serialización sin atributos para VOs con invariantes (commit a0c8ba4)**
Descubrimiento tecnico con fuentes verificadas: el *Contract Model de STJ* (disponible desde .NET 7) permite configurar la serializacion de campos privados desde infraestructura, sin contaminar la clase de dominio con `[JsonInclude]` o `[JsonConstructor]`. El equivalente del Fluent API de EF Core para serialización.

Patron canonico adoptado:
- Constructor privado parametrizado (para el factory)
- Constructor vacio privado (para STJ/Marten)
- Metodo `internal static ConfigurarSerializacion(DefaultJsonTypeInfoResolver)` en la misma clase — el mapping vive junto al objeto que describe

Verificado en .NET 10: `FieldInfo.SetValue` sobre campos `private readonly` de instancia funciona (la restriccion `readonly` aplica en compilacion, no en reflexion de instancia). Riesgo AOT documentado y aceptado.

**Actualizacion de agentes con las heuristicas aprendidas (commits 5ed646c, a5a0bba, e3a2329)**
Las sesiones de diseño se convirtieron en reglas operativas:
- `implementer`: sealed class + ConfigurarSerializacion obligatorio, encapsulamiento, constantes vs literales, evaluacion de factories multiples
- `reviewer`: checklist ampliado con cobertura de IEquatable y serializacion, 6 items nuevos en Definition of Done
- `test-writer`: testear value objects via `ToString()`, no via getters internos; seccion "Interfaz publica" como contrato vinculante
- `planner`: seccion "Interfaz publica" en issues de VOs complejos — el test-writer no adivina la API

**Issue #2 cerrado, PR #7 mergeado (18:38 — commit 1443575)**
El feature de value objects de franja temporal llego a main. 45 tests. El codigo final usa `sealed class` con encapsulamiento fuerte, serialización via Contract Model, `IgualdadTestBase<T>` para eliminar boilerplate en tests de igualdad, y `SubFranja` como clase intermedia unificando FranjaDescanso y FranjaExtra.

## Problemas encontrados

**PR #6: el pipeline aprobó tecnicamente, el diseño estaba roto**
La primera corrida del pipeline (07:16–07:40) produjo 45 tests verdes y creo PR #6. Pero la revision humana descubrio problemas de fondo que los tres agentes no detectaron:

- `HoraInicio`, `HoraFin`, `DiaOffsetFin`, `MinutosAbsolutoInicio/Fin` e `InferirDiaOffset` eran `public` — mecanica interna expuesta innecesariamente
- Los VOs eran `record` en lugar de `sealed class` — el `ToString()` sintetizado devolvia `FranjaDescanso { }` (campos privados, nada visible), `with {}` quedaba inutilizable
- Los literales `60` y `1440` dispersos en los calculos
- `Contiene` y `SeSolapan` eran metodos publicos de `FranjaOrdinaria` — la intencion era que fueran invariantes del constructor, no parte de la API
- Los tests verificaban propiedades via getters individuales en lugar de comportamiento via `ToString()`
- Los tests de excepciones no incluian `.WithMessage(...)` — solo verificaban el tipo

El reviewer automatico no los capturo porque no tenia reglas explicitas sobre encapsulamiento de VOs, ni sobre el criterio `record` vs `sealed class`, ni sobre testear por comportamiento.

**La segunda corrida del pipeline falló en el implementer (09:11)**
Ante la lista de correcciones que exigia el review del PR #6, se intento relanzar el pipeline para que el agente incorporara los cambios. El pipeline fallo en el stage 2. El agente que intenta corregir un PR con demasiados cambios simultaneos entra en un estado de confusion — arregla una cosa, rompe otra, pierde el hilo de los invariantes.

**El refactor del worktree tomo toda la tarde**
La conversion de `record` a `sealed class` implico reescribir constructores, factories, tests, configuracion de serializacion y la jerarquia de clases. Mas de 3 horas de trabajo manual (15:21–18:38) con varios commits de refactor sucesivos.

**Fixes de CI post-merge (19:04–19:49)**
El dominio `Programacion` necesitaba alinearse al estandar del `domain-scaffolder`. Al hacerlo aparecieron cuatro problemas encadenados: `global using Xunit` faltante en los tests, `global using Xunit` faltante en el template del scaffolder, sintaxis incorrecta de `dotnet test` en el workflow de CI, y `--collect` incompatible con MTP reemplazado por `dotnet-coverage`. Cuatro commits para resolverlo.

## Lo que descartamos

**Corregir el PR #6 iterativamente**
La decision de abandonar PR #6 y empezar desde cero con los nuevos conceptos fue la correcta. Cuando un review exige cambios en el tipo base (`record` → `sealed class`), en la visibilidad de multiples propiedades, en la jerarquia de herencia y en los tests simultaneamente, la deuda de navegacion mental del agente supera su capacidad de mantener coherencia. El resultado de intentar corregir fue un pipeline fallido y codigo a medias.

El abandon deliberado es una herramienta valida en vibe coding — no una derrota.

**`[JsonInclude]` y `[JsonConstructor]` en las clases de dominio**
Tecnicamente funciona. Pero contamina la clase con detalles de infraestructura. El Contract Model de STJ resuelve el mismo problema desde fuera de la clase, igual que EF Core configura persistencia sin tocar la entidad.

**Hora de corte por turno**
Se propuso que cada turno definiera a que hora empieza su "dia" (ej: 6AM para turnos nocturnos). Descartado: la particion por medianoche para recargos dominicales es responsabilidad de CalculoHoras segun la legislacion colombiana. Agregar esa logica a Programacion seria violar la autonomia de dominios.

**Campo "tipo" en las tuplas del comando**
Propuesta: `{ inicio, fin, tipo: "ordinaria"|"descanso"|"extra" }`. Descartada: el tipo es implicito en la propiedad del arreglo donde vive la tupla. Un campo explicito seria redundante y abriria la puerta a inconsistencias (tipo "descanso" en el arreglo de ordinarias).

**Publicar TurnoCreado a Service Bus**
No hay consumidores externos del evento todavia. Publicarlo seria ruido — un evento sin destinatario en un sistema event-driven es infraestructura desperdiciada. Se publica cuando haya consumidores.

**Result pattern para errores del factory**
`AggregateException` es suficiente: el handler de Wolverine ya maneja excepciones, y un Result pattern agregaria una capa de wiring sin ganancia real en este contexto.

## Aprendizajes

1. **En vibe coding, hay un umbral donde corregir un PR cuesta mas que descartarlo.** Cuando el review exige cambios estructurales (tipo base, visibilidad, jerarquia) y cambios en tests simultaneamente, el agente pierde coherencia. El punto de corte no es cuantitativo — es cuando los cambios son interdependientes y el agente no puede mantener el invariante global mientras corrige partes. La senal: el pipeline falla en el implementer al intentar aplicar el feedback.

2. **"Tests verdes" no es lo mismo que "buen disenyo".** El pipeline puede producir 45 tests verdes con codigo que viola encapsulamiento, usa el tipo equivocado y expone mecanica interna. La diferencia: los tests estaban escritos para verificar los getters publicos equivocados, no el comportamiento. Un reviewer que no tiene reglas explicitas sobre diseño de VOs no puede detectar lo que no sabe que debe buscar.

3. **El criterio `record` vs `sealed class` es mutabilidad, no complejidad.** La intuicion de "si tiene reglas, usa clase" es imprecisa. Un VO puede tener validaciones sofisticadas y seguir siendo un `record` inmutable si sus datos son publicamente accesibles. El `sealed class` es necesario cuando el VO encapsula estado privado — porque `record` con campos privados produce un `ToString()` vacio y un `with {}` inutilizable.

4. **La seccion "Interfaz publica" en los issues de VOs es un contrato, no documentacion.** Sin ella, el test-writer adivina que propiedades son publicas y tiende a sobreexponer. Con ella, tiene un contrato vinculante: solo puede poner `public` lo que aparece ahi. Esto previene exactamente el tipo de encapsulamiento roto que revelo el review del PR #6.

5. **El mapping de serializacion pertenece al mismo archivo que el VO.** Si vive en infraestructura, se desincroniza cuando cambia el VO. Si vive junto al VO como `internal static ConfigurarSerializacion(...)`, el compilador fuerza la actualizacion — no puede haber un campo nuevo sin que alguien actualice el mapping.

6. **Las heuristicas de disenyo deben estar en los agentes, no solo en ADRs.** El ADR-0015 documenta la decision para humanos. Las reglas en el `implementer` y el `reviewer` son las que realmente cambian el comportamiento del pipeline. Un ADR sin su contraparte en los agentes es un documento que se lee una vez y no se aplica.
