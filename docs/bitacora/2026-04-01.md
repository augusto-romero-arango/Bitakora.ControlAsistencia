## Problemas encontrados

**Nombre del storage account ya existia globalmente en Azure (19:48)**
El nombre `stcadevfunc` estaba tomado. Azure requiere nombres de storage accounts globalmente unicos. Se agrego un sufijo `random_string` de 6 caracteres. Un minuto despues se fue mas lejos y se decidio que cada Function App tuviera su propio storage account.

**Publish profile vs Service Principal (20:30)**
El scaffolder generaba un secret `AZURE_FUNC_PROGRAMACION_PUBLISH_PROFILE` por Function App. Mirando el repo hermano ControlPlane, este usaba un Service Principal unico con `azure/login@v2`. Se migro al patron correcto.

**Function App vacia = "malformed content" (21:11)**
Azure rechazaba el deploy de una Function App sin ninguna funcion con un error "malformed content". La solucion inmediata fue crear un `HealthCheck.cs` como HTTP trigger minimo. La causa raiz (falta de `FUNCTIONS_WORKER_RUNTIME`) se descubrio al dia siguiente.

**Deploy que se extiende al dia siguiente (commits del 2 de abril, 07:33-08:51)**
A las 21:11 el dia termino con el deploy aun sin funcionar. A la manana siguiente se encontraron tres causas raiz adicionales: (1) faltaba `FUNCTIONS_WORKER_RUNTIME=dotnet-isolated` en el Terraform, (2) la configuracion de publish no alineaba con ControlPlane (`-r linux-x64 --self-contained false`), (3) el plan de hosting Consumption Y1 no soporta .NET 10 - minimo B1. Cuatro commits mas para dejarlo funcional.

## Lo que descartamos

**Lista de dominios heredada del repo anterior**
Del CLAUDE.md se elimino `Dominios: CalculoHoras, Depuración, Programación, Empleados`. Estos nombres venian de ControlAsistencias y necesitaban revalidarse con el modelo de dominio actual antes de imponerse como verdad.

**"El turno es un objeto, no una referencia"**
Se elimino este principio de modelado del CLAUDE.md. Era una decision de diseno de agregados demasiado especifica para documentar antes de haber escrito una sola linea de dominio.

**Storage account compartido entre Function Apps**
El primer enfoque era un storage account compartido. A los diez minutos se cambio a uno dedicado por Function App. Razon: cap. 8 de "Beginning Azure Functions" recomienda storage dedicado para evitar contenciones de tabla en deployments concurrentes.

**ADR-0004 original (topic por dominio) - quedara obsoleto al dia siguiente**
El modelo de Service Bus de este primer dia usaba un topic por dominio productor con subscriptions SQL filter. Al dia siguiente se descubrio que esto ocultaba la complejidad operativa y se invirtio a un topic por tipo de evento.

## Aprendizajes

1. **Plan Consumption Y1 de Azure Functions no soporta .NET 10.** El plan minimo funcional es B1 (Basic dedicado). Si la Function App arranca con 503 intermitentes o no registra funciones, el plan de hosting es el primer sospechoso.

2. **`FUNCTIONS_WORKER_RUNTIME=dotnet-isolated` es obligatorio en app settings de Terraform.** Sin esta variable, Azure no sabe como ejecutar el worker y rechaza el despliegue con errores oscuros como "malformed content".

3. **`-r linux-x64 --self-contained false` es la configuracion correcta de publish para .NET 10 en Azure Functions Linux.** Self-contained true genera un ejecutable que conflictua con el runtime del host.

4. **Validar el artefacto antes de hacer push al deploy.** El workflow evolucionado verifica que existan `host.json`, `functions.metadata` y el DLL del proyecto antes de intentar el deploy, fallando rapido con un mensaje claro.

5. **El repo hermano como oraculo.** Cuando algo no funciona, comparar con ControlPlane (que si funciona en produccion) fue consistentemente la forma mas rapida de encontrar la diferencia.

6. **Limpiar decisiones prematuras es trabajo real.** Los 12 minutos del segundo commit (e750b3d) evitaron cargar con restricciones de modelado no validadas durante semanas.
