## Problemas encontrados

**Incidente de produccion: MessageLockLost en cascada (21:10)**
La alerta de excepciones (ADR-0009) salto con 8 fallos en 22 segundos en `AsignarTurnoCuandoProgramacionTurnoDiarioSolicitada` (dominio ControlHoras). El agente `bug-investigator` — recien desplegado horas antes — investigo el primer caso real de su existencia.

Causa raiz: la combinacion de tres factores en `host.json`:

```
prefetchCount: 10  +  maxConcurrentCalls: 1  +  lock duration: 1 minuto (default)
```

El flujo del desastre: los smoke tests publicaron eventos al topic. La Function App arranco en cold start (~79 segundos). El SDK bajo 10 mensajes de golpe con sus locks corriendo. Mientras procesaba el primero (79s), los otros 9 perdieron el lock. Service Bus los devolvio a la subscription. Cuando la funcion intento `CompleteMessageAsync` en el mensaje #2, el lock ya no existia: `MessageLockLost`. El catch block intento `DeadLetterMessageAsync` — que tambien falla sin lock vigente. Una excepcion por cada mensaje afectado.

Los mensajes no se perdieron: Service Bus los re-entrego automaticamente y se procesaron exitosamente en el segundo intento (~282ms, Function App ya caliente).

**Defecto secundario en el catch block de FunctionEndpoint**
El bloque `catch` no distinguia errores de infraestructura de errores de negocio. Intentar dead-letter cuando el lock ya expiro es una operacion que siempre va a fallar y duplica las excepciones en los logs. Fix: detectar `ServiceBusException` con `Reason == MessageLockLost` y solo loggear advertencia, sin operar sobre el mensaje. El issue #48 lo corrigio antes del cierre del dia (cerrado el 11 a las 10:26).

**Hipotesis inicial descartada rapidamente**
La primera hipotesis fue que el TTL de 5 minutos recien configurado en la subscription `smoke-tests` estaba expirando los mensajes. Se descarto: cada subscription mantiene su propia copia del mensaje — el TTL de una no afecta a otra. El diagnostico correcto costo menos de 20 minutos con App Insights.

## Lo que descartamos

**Azure MCP Server oficial para acceso a App Insights**
El servidor MCP de Microsoft (febrero 2026) solo expone "list code optimization recommendations". No ejecuta queries KQL. Insuficiente para debugging real.

**MCPs de terceros para KQL**
`furoTmark/application-insights-mcp`: repo con README vacio. `4R9UN/mcp-kql-server`: apunta a ADX clusters, no App Insights directamente. Inmaduros, no documentados.

**Pipeline bash con worktree/tmux para debugging**
El debugging es fundamentalmente conversacional: el humano necesita validar hipotesis y dar contexto de negocio. Un script con stages automatizados no es el formato correcto. Un agente de Claude es el formato natural.

**Queries KQL generadas ad-hoc por el agente**
Con el daily cap de 0.5 GB del ADR-0009, queries abiertas podrian ser costosas o ineficientes. Queries predefinidas con `take N` y `ago(Xh)` son auditables, conservadoras, y suficientes para el 90% de los casos.

**TTL global a nivel de topic en Service Bus**
Afectaria a todas las subscriptions del topic, incluyendo las de dominio que necesitan retencion larga. El TTL es responsabilidad de quien declara la subscription.

**Descubrimiento dinamico de dominios para el workflow global de smoke tests**
Buscar `tests/*SmokeTests/*.csproj` en runtime es menos predecible y no obliga al scaffolder a mantener el registro actualizado. Un archivo JSON explicito da control total.

**Configurar `lock_duration` en Terraform (por ahora)**
Con `prefetchCount: 0`, el lock duration default de 1 minuto es suficiente. El SDK renueva el lock automaticamente hasta por 5 minutos con `maxAutoLockRenewalDuration`. Si en el futuro se sube prefetchCount, se revisara en conjunto. ADR-0017 documenta la condicion.

## Aprendizajes

1. **`prefetchCount` alto solo tiene sentido cuando `maxConcurrentCalls` es alto.** Con procesamiento secuencial, prefetch baja N mensajes con sus locks corriendo pero solo procesa 1 a la vez. Los N-1 restantes tienen el reloj corriendo sin ser atendidos. La guia de Microsoft es `prefetchCount = maxConcurrentCalls x 20`. Con `maxConcurrentCalls: 1`, el valor correcto es 0 o 1.

2. **Un cold start largo puede ser mas letal que un procesamiento lento.** 79 segundos de cold start con mensajes pre-descargados y lock de 1 minuto es una combinacion explosiva. El cold start no es solo un problema de latencia — puede causar perdida de locks y cascada de errores.

3. **No se puede operar un mensaje sin lock vigente.** Intentar `DeadLetterMessageAsync` despues de `MessageLockLost` siempre va a fallar. El catch block correcto detecta el tipo de excepcion antes de decidir la accion: si el lock se perdio, solo se loggea — Service Bus ya se encargo de devolver el mensaje.

4. **Construir la herramienta de debugging mientras hay un incidente activo es la validacion mas efectiva posible.** El `bug-investigator` investigó su primer caso real el mismo dia que se desplegó. El diagnostico fue correcto, los fixes surgieron de la investigacion, y el ADR quedo escrito con datos reales del incidente en lugar de hipoteticos.

5. **Los pipelines de infraestructura y tooling necesitan el mismo patron de aislamiento que TDD.** Worktree + tmux + no bloquear la terminal no es solo conveniencia — permite correr multiples pipelines en paralelo sin interferencia. La migracion del pipeline IaC completó el conjunto.
